# 実装計画書: 2ch/5chまとめ風 マルチエージェントLLM議論ジェネレーター

## 概要

複数のLLMエージェント（各々に独自のペルソナを付与）がユーザー指定のテーマについて掲示板風の議論を自律的に行い、その議論履歴を2ch/5chまとめサイト風のHTMLデザインで表示するWebアプリケーション。Microsoft AutoGen（v0.7.x AgentChat API）を用いて独立した複数エージェントを構築し、GradioのカスタムHTML機能（v6.x `gr.HTML`）でまとめサイト風のリッチな出力表示を実現する。出力は画面表示のほか、`.txt`・`.json`・`.html` ファイルとしてダウンロードも可能。

---

## UI選定: Gradio（オリジナルUIではなくGradio）

**選定理由:** Gradio v6.x は `gr.HTML` コンポーネントでカスタムHTML・CSS・JavaScriptを自由にレンダリングでき、まとめサイト風のデザインを完全に再現できる。オリジナルUI（Flask/FastAPI + フロントエンド）を構築する場合、フロントエンド開発（HTML/CSS/JS）とバックエンドAPI設計の両方が必要になり、自力でコードを書けないユーザーにとっては保守コストが高すぎる。Gradioであれば Python単一ファイルでUI定義・バックエンドロジック・ファイル出力まで完結し、`launch()` 一発で起動できる。カスタムCSSとカスタムHTMLテンプレートを組み合わせることで、まとめサイト風のデザインは十分に実現可能。

---

## 要件定義

### 機能要件

**FR-1:** ユーザーがテーマ（スレッドタイトル）、議論の方向性、補足テキスト、画像、ファイルを入力できること。

**FR-2:** 複数のLLMエージェント（3～10体を想定）が、それぞれ独立したペルソナ（名前、口調、立場）を持ち、テーマについて掲示板風の議論を行うこと。

**FR-3:** 各エージェントは独立したLLMクライアントとして動作し、1つのLLMに複数人格を演じさせる偽装方式ではないこと。

**FR-4:** 対応LLMプロバイダは OpenAI API、Google Gemini（OpenAI互換API経由）、Ollama（ローカルLLM）、LM Studio（OpenAI互換API経由）の4種。

**FR-5:** 議論結果を2ch/5chまとめサイト風のHTML/CSSテンプレートで表示すること（レス番号、名前欄、ID、日時、本文のフォーマット）。

**FR-6:** 議論結果を `.txt`（プレーンテキスト）、`.json`（構造化データ）、`.html`（まとめサイト風スタンドアロンHTML）としてダウンロードできること。

**FR-7:** 議論のリアルタイム進行をストリーミング表示できること。

### 非機能要件

**NFR-1:** Mac mini M4（Apple Silicon, 24GB RAM, macOS Tahoe）で動作すること。GPU指定は `mps` または `cpu`。

**NFR-2:** Python 3.x、文字コード UTF-8、PEP 8準拠、日本語コメント。

**NFR-3:** パッケージマネージャーは `uv` を使用（`pip` 不使用）。

**NFR-4:** Gradio `launch()` 時に `server_name="127.0.0.1"` をデフォルトにすること。

## 設定項目一覧（ユーザーがUIから操作可能な項目）

以下の設定項目をUI上に配置する。

**基本設定:**
`スレッドタイトル`（必須・テキスト）、`議論の方向性・テーマ説明`（任意・テキストエリア）、`添付ファイル`（任意・画像/テキストファイル）。

**エージェント設定:**
`エージェント数`（スライダー: 3～10、デフォルト5）、`ペルソナプリセット`（ドロップダウン: 「バランス型」「賛成多数」「反対多数」「カオス」「専門家会議」から選択）、`カスタムペルソナJSON`（上級者向け・任意テキストエリア、プリセットの上書き用）。

**LLM設定:**
`デフォルトLLMプロバイダ`（ドロップダウン: OpenAI / Gemini / Ollama / LM Studio）、`モデル名`（テキスト、デフォルト `gpt-4o`）、`APIキー`（パスワード入力欄、環境変数からの読み込みも対応）、`ベースURL`（Ollama/LM Studio用、デフォルト空欄）、`Temperature`（スライダー: 0.0～2.0、デフォルト0.9）。

**議論設定:**
`議論ラウンド数`（スライダー: 3～30、デフォルト10）、`1レスあたりの最大文字数`（スライダー: 50～500、デフォルト200）、`議論スタイル`（ドロップダウン: 「雑談」「真面目な議論」「煽り合い」「ネタ」）、`まとめ人（>>1）の有無`（チェックボックス、デフォルトON）。

**出力設定:**
`出力フォーマット`（チェックボックスグループ: 画面表示 / .txt / .json / .html、デフォルトは全チェック）。

# LM Studio (OpenAI互換API経由)

    model="local-model",
    base_url="http://localhost:1234/v1",

### フェーズ3: マルチエージェント議論エンジン

**ステップ 3-1: 議論実行モジュール**
対象: `src/matome_generator/agents/discussion.py`
操作: AutoGen の `SelectorGroupChat` を使い、複数の `AssistantAgent`（各ペルソナに対応）によるグループチャットを構築する `DiscussionEngine` クラスを実装。`SelectorGroupChat` を選択する理由は、ラウンドロビンでは機械的な順番になってしまうが、`SelectorGroupChat` は文脈に応じて次の発言者を自動選択でき、より自然な掲示板の会話が再現できるため。ただし、全員が均等に発言する保証がないため、`selector_prompt` をカスタマイズして「まだ発言が少ない参加者を優先せよ」「同じ人が連続で発言しすぎないようにせよ」という指示を含める。

主要なメソッド: `async def run_discussion(topic, config) -> list[Message]`（議論全体を実行し、メッセージ一覧を返す）、`async def run_discussion_stream(topic, config) -> AsyncGenerator[Message]`（ストリーミング版）。

終了条件には `MaxMessageTermination(count=ラウンド数 * 2)` を使用（安全マージンとして倍数を設定し、議論が暴走しないようにする）。加えて、`>>1` のまとめ人エージェント（オプション）を最後に発言させて「以上で議論終了」と宣言させる仕組みも入れる。

各エージェントには異なるLLMクライアントを割り当てることができる設計にする（例: エージェントAはGPT-4o、エージェントBはGemini）。ただし、デフォルトでは全エージェントが同じプロバイダ・モデルを使用する。

依存: ステップ 2-1, 2-2
リスク: 高（AutoGenのバージョン変更によるAPI非互換のリスク。`SelectorGroupChat` のセレクタモデル自体にもLLMコストがかかる点に注意。セレクタ用に安価なモデルを使う設計にする）

**重要な設計判断 — `SelectorGroupChat` vs `RoundRobinGroupChat`:**

`RoundRobinGroupChat` は実装がシンプルだが、A→B→C→D→E→A→… という機械的な順番になり、掲示板の自然な会話感が出ない。`SelectorGroupChat` は「次に誰が喋るか」をLLMが文脈に基づいて決めるため、「>>3に反論したい人が割り込む」「話題に詳しい人が反応する」といった自然な流れが生まれる。コストは若干上がるが、出力品質を重視して `SelectorGroupChat` を採用する。セレクタ用LLMは安価なモデル（`gpt-4o-mini` 等）を使い、コストを抑制する。

**ステップ 3-2: メッセージデータ構造**
対象: `src/matome_generator/agents/discussion.py` 内
操作: AutoGenの `TaskResult.messages` から受け取った生メッセージを、まとめ表示用の独自データクラス `ThreadMessage` に変換する。`ThreadMessage` は `res_number`（レス番号）、`name`（名前欄）、`trip`（トリップ）、`uid`（ID文字列、ペルソナ毎にランダム生成）、`timestamp`（擬似的な投稿日時）、`body`（本文）を持つ。
依存: ステップ 3-1
リスク: 低

**ステップ 3-3: 議論エンジンの単体テスト**
対象: `tests/test_discussion.py`
操作: モックLLMクライアントを使い、`DiscussionEngine` がエージェントを正しく構築すること、メッセージが `ThreadMessage` に正しく変換されること、ラウンド数制限で議論が終了することをテスト。実際のAPI呼び出しは行わない（コスト回避）。
依存: ステップ 3-1, 3-2
リスク: 低

### フェーズ4: まとめ風レンダリング

**ステップ 4-1: CSSスタイル定義**
対象: `src/matome_generator/rendering/matome_css.py`
操作: 2ch/5chまとめサイト風のCSSをPython文字列定数として定義。背景色（`#efefef`）、レスボックス（白背景・薄いボーダー）、名前欄（緑色 `#117743` のボールド）、レス番号（青リンク色）、IDとタイムスタンプ（グレー小文字）、アンカー（`>>` 参照を赤リンク色に）、スレッドタイトル（大きめフォント・中央寄せ）、全体のフォント（`"Meiryo", "ヒラギノ角ゴ", sans-serif`）のスタイルを定義。レスポンシブ対応も含む。
依存: なし
リスク: 低

CSSの概要設計:
```
.matome-container: 背景 #efefef, max-width 800px, margin 0 auto, padding 16px
.thread-title: font-size 1.4em, font-weight bold, color #cc3300, border-bottom 2px solid #cc3300
.res-box: background #fff, border 1px solid #ddd, margin 8px 0, padding 10px
.res-header: font-size 0.85em, color #666
.res-number: color #0000ff, font-weight bold
.res-name: color #117743, font-weight bold
.res-trip: color #999
.res-uid: color #888
.res-timestamp: color #888
.res-body: margin-top 6px, line-height 1.6, word-break break-all
.res-anchor: color #ff0000
```

**ステップ 4-2: HTMLテンプレート生成**
対象: `src/matome_generator/rendering/matome_html.py`
操作: Jinja2テンプレートを使い、`list[ThreadMessage]` からまとめサイト風HTMLを生成する関数 `render_matome_html(title, messages, css) -> str` を実装。Gradio `gr.HTML` に渡すインライン用HTML（`<style>` + `<div>`）と、スタンドアロンHTMLファイル用（完全なHTML5ドキュメント）の2パターンを生成できるようにする。本文中の `>>{数字}` パターンをアンカーリンク（`<span class="res-anchor">&gt;&gt;{数字}</span>`）に自動変換する処理も含む。
依存: ステップ 4-1
リスク: 低

レス1件分のHTMLテンプレート:
```html
<div class="res-box" id="res-{{ msg.res_number }}">
  <div class="res-header">
    <span class="res-number">{{ msg.res_number }}</span>：
    <span class="res-name">{{ msg.name }}</span>
    <span class="res-trip">{{ msg.trip }}</span>
    {{ msg.timestamp }}
    ID:<span class="res-uid">{{ msg.uid }}</span>
  </div>
  <div class="res-body">{{ msg.body_html }}</div>
</div>
```

**ステップ 4-3: エクスポーター**
対象: `src/matome_generator/rendering/exporters.py`
操作: `export_txt(title, messages) -> str`（プレーンテキスト形式）、`export_json(title, messages) -> str`（JSON形式、各メッセージの全フィールドを含む辞書のリスト）、`export_html(title, messages) -> str`（スタンドアロンHTML5ファイル）の3関数を実装。各関数はファイル内容の文字列を返す。Gradio側でファイル書き出し・ダウンロードリンク化を行う。
依存: ステップ 4-1, 4-2
リスク: 低

**ステップ 5-1: UIレイアウト定義**
対象: `src/matome_generator/app.py`
操作: `gr.Blocks` を使い、以下のレイアウトを構築。

画面構成（上から順に）:

**ヘッダーエリア:** アプリタイトル「まとめジェネレーター」。

**入力タブ（`gr.Tab("入力")`）:**
左カラム — `スレッドタイトル`（`gr.Textbox`）、`議論の方向性`（`gr.Textbox`, lines=3）、`添付ファイル`（`gr.File`, file_types=["image", "text"]）。
右カラム — `エージェント数`（`gr.Slider`）、`ペルソナプリセット`（`gr.Dropdown`）、`カスタムペルソナJSON`（`gr.Code`, language="json", visible=False、プリセット「カスタム」選択時に表示）、`議論ラウンド数`（`gr.Slider`）、`1レスあたりの最大文字数`（`gr.Slider`）、`議論スタイル`（`gr.Dropdown`）、`まとめ人の有無`（`gr.Checkbox`）。

**LLM設定タブ（`gr.Tab("LLM設定")`）:**
`デフォルトLLMプロバイダ`（`gr.Dropdown`）、`モデル名`（`gr.Textbox`）、`APIキー`（`gr.Textbox`, type="password"）、`ベースURL`（`gr.Textbox`）、`Temperature`（`gr.Slider`）。

**実行ボタン:** `gr.Button("議論を開始", variant="primary")`、`gr.Button("中止", variant="stop")`。

**出力エリア:**
`まとめプレビュー`（`gr.HTML`, label="まとめプレビュー"）— ここにまとめ風HTMLをリアルタイム表示。
`ダウンロード`（`gr.File`, label="ダウンロード", file_count="multiple"）— .txt, .json, .html ファイル。

依存: ステップ 1-2
リスク: 中（Gradioのレイアウトは細かい挙動がバージョン依存。`gr.HTML` の更新タイミングに注意）

**ステップ 5-2: イベントハンドラ接続**
対象: `src/matome_generator/app.py`
操作: 「議論を開始」ボタンのクリックイベントに非同期ハンドラを接続。ハンドラは以下の処理を行う: (1) UI入力からの `DiscussionConfig` 構築、(2) `DiscussionEngine` の初期化、(3) `run_discussion_stream()` でストリーミング実行、(4) 各メッセージ受信時に `render_matome_html()` を呼んで `gr.HTML` を更新（`yield` で逐次返却）、(5) 完了後に `.txt`, `.json`, `.html` ファイルを `outputs/` に書き出し、`gr.File` にパスを返す。

ストリーミング更新は Gradio の `generator` パターンを使う: ハンドラを `async generator`（`yield`）にし、メッセージが増えるたびにHTMLを丸ごと再レンダリングして `yield` する。

「中止」ボタンは `ExternalTermination` を使って議論を安全に停止する。

依存: ステップ 3-1, 4-2, 4-3, 5-1
リスク: 高（非同期ストリーミングとGradioのイベントループの統合が複雑。AutoGenは `asyncio` ベース、Gradioも内部で `asyncio` を使うため互換性は良いが、エラーハンドリングを入念に行う必要がある）

**ステップ 5-3: エントリポイント**

操作: `if __name__ == "__main__":` ブロックで `demo.launch(server_name="127.0.0.1")` を呼ぶ。`pyproject.toml` の `[project.scripts]` にもエントリポイントを定義: `matome = "matome_generator.app:main"`。

リスク: 低

## リスクと対策

**リスク1: AutoGenのバージョン非互換**
`autogen-agentchat` は活発に開発中で、0.7.x → 0.8.x でAPIが変わる可能性がある。対策として `pyproject.toml` でバージョンを `>=0.7.0,<0.8.0` にピン止めする。

**リスク2: LLMが掲示板風の口調を維持しない**
エージェントが途中で丁寧な口調に戻ったり、メタ的な発言をする可能性がある。対策としてシステムプロンプトに「あなたはAIではなく掲示板の住人である。メタ的な発言は禁止」という強い指示を入れ、各ラウンドの冒頭にもリマインドを挟む。

**リスク3: Gemini/Ollama の OpenAI互換APIの不安定さ**
一部のレスポンスフォーマットが微妙に異なる場合がある。対策として `model_factory.py` 内でプロバイダごとにレスポンスのバリデーション・正規化処理を行う。

**リスク4: 議論が暴走（無限ループ、話題の逸脱）**
対策として `MaxMessageTermination` で上限を設定し、`ExternalTermination` で手動中止も可能にする。

**リスク5: APIコストの増大**
`SelectorGroupChat` のセレクタLLM呼び出しが各ラウンドで追加コストになる。対策としてセレクタ用モデルを `gpt-4o-mini` 等の安価なモデルに固定する設定をデフォルトにする。
