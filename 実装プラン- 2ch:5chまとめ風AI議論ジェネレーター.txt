# 実装計画書: 2ch/5chまとめ風コンテンツ自動生成アプリケーション

## 概要

ユーザーが入力したテーマ・議論方針に基づき、複数の独立したLLMエージェントが2ちゃんねらー風に議論を行い、その議論ログを別のAIエージェントが「まとめサイト風」に編集・出力するWebアプリケーションです。マルチエージェント基盤にはMicrosoft AutoGen AgentChat（`autogen-agentchat`）の`RoundRobinGroupChat`を採用し、UIにはGradio（`gr.HTML`コンポーネントによるまとめ風表示）を使用します。

## UI判断: Gradio vs オリジナルUI

Gradioを採用します。理由は以下の通りです。

ユーザーがコードを書けない前提のため、Gradioならバックエンド（Python）だけで完結し、HTML/CSS/JSの個別管理が不要です。`gr.HTML`コンポーネントでカスタムHTML+CSSを描画できるため、まとめサイト風のリッチな表示も実現可能です。ファイルダウンロード（`gr.File`）、プルダウン（`gr.Dropdown`）、チェックボックス（`gr.CheckboxGroup`）、スライダー（`gr.Slider`）等の入力UIが即座に使えます。また、`launch(server_name="127.0.0.1")`の規約にも適合します。

## 要件定義

### 機能要件

**FR-1**: ユーザーがテーマ（テキスト）、議論方向性、議論トーン、会話数を入力できること。

**FR-2**: 複数の独立LLMエージェントが、それぞれ異なるペルソナを持ち、2ちゃんねらー風の口調で議論を行うこと。

**FR-3**: 各エージェントが独立したLLMインスタンス（異なるプロバイダー可）であること。単一LLMの偽装複数ではないこと。

**FR-4**: 議論終了後、別のまとめAIエージェントが議論ログを2ch/5chまとめサイト風に整形すること。

**FR-5**: スレッドタイトルもAIが自動生成すること。

**FR-6**: 出力形式として、画面上のまとめ風HTML表示、`.txt`ファイル、`.json`ファイルのエクスポートが可能であること。

**FR-7**: 対応LLMプロバイダー: OpenAI API、Google Gemini（OpenAI互換API経由）、Ollama（ローカル）、LM Studio（OpenAI互換API経由）。

**FR-8**: 企業APIへの連続アクセス制限を防ぐため、リクエスト間にウェイトタイム（秒数指定可能）を設けること。

**FR-9**: テキスト・画像・ファイルの入力に対応すること（画像・ファイルはテーマ説明の補助として添付）。

### 非機能要件

**NFR-1**: Mac mini M4（Apple Silicon, 24GB）、macOS 26.3（Tahoe）で動作すること。

**NFR-2**: Python 3.10以上（AutoGenの要件）。

**NFR-3**: パッケージマネージャーは`uv`を使用。

**NFR-4**: PEP 8準拠、日本語コメント、Absolute Import。

### Phase 2: LLMクライアントファクトリーとレートリミッター

**ステップ 2-1: レートリミッター実装**

- 対象: `src/utils/rate_limiter.py`
- 内容: `asyncio.sleep`ベースのシンプルなレートリミッタークラスを実装する。連続するAPIコール間に指定秒数の待機を挿入する。
- 理由: 企業APIの連続アクセス制限を防止する。AutoGenのエージェント呼び出しの前後で呼ばれる仕組みとする。
- 依存関係: ステップ 1-2
- リスク: 低

クラス設計:

```python
class RateLimiter:
    """APIリクエスト間のウェイトタイムを管理するクラス"""
    def __init__(self, wait_seconds: float = 1.0):
        ...
    async def wait(self) -> None:
        """前回のリクエストからwait_seconds秒以上経過するまで待機する"""
        ...
```

**ステップ 2-2: LLMクライアントファクトリー実装**

- 対象: `src/models/client_factory.py`
- 内容: プロバイダー名とモデル名を受け取り、対応するAutoGenのモデルクライアントを返すファクトリー関数を実装する。
- 理由: エージェント生成時にプロバイダーの違いを吸収し、統一的なインターフェースを提供する。
- 依存関係: ステップ 1-3, ステップ 2-1
- リスク: 中（Gemini互換APIの挙動、Ollamaローカルサーバーの接続確認が必要）

プロバイダー別の生成ロジック:

`openai`: `OpenAIChatCompletionClient(model=model_name, api_key=key)` を使用。

`gemini`: `OpenAIChatCompletionClient(model=model_name, api_key=key)` を使用。Gemini は OpenAI互換APIを提供しているため、`autogen-ext[openai]` で対応可能。必要に応じ `model_info` パラメータで `ModelInfo(vision=True, function_calling=True, json_output=True, family="unknown", structured_output=True)` を設定する。

`ollama`: `OllamaChatCompletionClient(model=model_name, host=ollama_base_url)` を使用。`autogen-ext[ollama]` が必要。

`lmstudio`: `OpenAIChatCompletionClient(model=model_name, base_url=lmstudio_base_url, api_key="lm-studio")` を使用。LM StudioはOpenAI互換APIを提供するため、`base_url`を差し替えるだけで対応可能。`api_key`にはダミー値を渡す。

返り値の型は `autogen_core.models.ChatCompletionClient`。

### Phase 3: ペルソナ定義とエージェント構築

**ステップ 3-1: ペルソナ定義**

- 対象: `src/agents/persona.py`
- 内容: 2ちゃんねらー風のペルソナ（名前、口調、性格）を定義するデータクラスと、議論トーンに応じてペルソナ群を自動生成する関数を実装する。
- 理由: 各エージェントに個性を持たせ、リアルな掲示板風の多様な会話を実現する。
- 依存関係: ステップ 1-2
- リスク: 低

ペルソナのデータ構造:

```python
@dataclass
class Persona:
    name: str           # 名無し風の表示名（例: 「名無しさん＠お腹いっぱい。」）
    display_id: str     # ID表示（例: 「ID:a3Kx9pB0」）
    personality: str    # 性格の概要
    speech_style: str   # 口調の説明
    stance: str         # 議論への立場（賛成寄り、批判的、中立、etc.）
```

ペルソナプール（組み合わせ用のテンプレート）:

名前プール: 「名無しさん＠お腹いっぱい。」「名無しさん＠実況は禁止です」「風吹けば名無し」「名無しさん＠おーぷん」「以下、名無しに変わりましてVIPがお送りします」など。

口調プール: 「丁寧語で論理的に話す」「草を多用するノリの軽い話し方」「煽り気味で断定的」「やたら長文で語る古参風」「短文でツッコミを入れる」「方言混じり」「やる気のない脱力系」「知識マウントを取りたがる」「すぐ話を脱線させる」など。

ペルソナ生成関数 `generate_personas(count: int, tones: list[str]) -> list[Persona]`:
- `tones` パラメータ（ユーザーが選んだ議論トーン）に応じ、ペルソナの立場分布を調整する。
- 例: 「白熱」が選択されていれば、対立する立場のペルソナを多くする。
- 例: 「煽り」が選択されていれば、煽り口調のペルソナを追加する。
- IDは `random`で疑似ランダム英数字8文字を生成する。

**ステップ 3-2: 各エージェント用システムプロンプト生成**

- 対象: `src/agents/persona.py` 内の関数
- 内容: ペルソナ情報とテーマを受け取り、AutoGenの `AssistantAgent` に渡す `system_message` 文字列を生成する関数を実装する。
- 依存関係: ステップ 3-1
- リスク: 低

システムプロンプトの基本テンプレート:

```
あなたは日本の匿名掲示板（2ちゃんねる/5ちゃんねる）の住人です。
以下の設定に従い、掲示板風の投稿を1レスだけ書いてください。

【あなたの設定】
- 表示名: {persona.name}
- 性格: {persona.personality}
- 口調: {persona.speech_style}
- 立場: {persona.stance}

【議論のテーマ】
{theme}

【議論の補足情報】
{context}

【ルール】
- 1回の投稿で1レスのみ書く
- 他の人のレスに反応しても良い（アンカー >> を使う）
- 2ch/5ch風の自然な会話をする
- 顔文字やAA（アスキーアート）を適度に使って良い
- 「ワロタ」「草」「〜だろ」「〜じゃね？」などの掲示板特有の表現を使う
- レスの長さは1〜5行程度にする（長文キャラ以外）
- 他の住人と議論をする。前のレスの流れに沿う
```

**ステップ 3-3: 議論エージェント群の構築と実行**

- 対象: `src/agents/discussion.py`
- 内容: ペルソナ群からAutoGen `AssistantAgent` を生成し、`RoundRobinGroupChat` で議論を実行する関数を実装する。
- 理由: AutoGen AgentChatの `RoundRobinGroupChat` を使うことで、各エージェントがラウンドロビン方式で順番に発言する。会話数は `MaxMessageTermination` で制御する。
- 依存関係: ステップ 2-2, ステップ 3-2
- リスク: 高（マルチエージェント会話の品質、非同期処理、レート制限のタイミング）

実装のポイント:

```python
from autogen_agentchat.agents import AssistantAgent
from autogen_agentchat.teams import RoundRobinGroupChat
from autogen_agentchat.conditions import MaxMessageTermination
```

各エージェントは独立した `AssistantAgent` として生成する。プロバイダーが同じ場合でも、`model_client` はエージェントごとに個別のインスタンスを作る（独立性の確保）。

`MaxMessageTermination(max_messages=conversation_count)` でユーザー指定の会話数で自動停止する。

ウェイトタイムの挿入方法: AutoGenの `RoundRobinGroupChat` は内部的に各エージェントの応答を順番に取得する。レート制限のために、`run_stream` のイテレーション中で `rate_limiter.wait()` を呼ぶラッパー関数を実装する。具体的には、`team.run_stream(task=...)` から得られるメッセージをイテレートし、`TextMessage` が得られるたびに `await rate_limiter.wait()` を行う。ただし、`RoundRobinGroupChat` の内部的にはエージェント呼び出し時にwaitを挟めないため、**カスタムエージェントクラス**で`on_messages`をオーバーライドし、応答前にwaitを入れるアプローチが最も確実である。

カスタムエージェントクラス:

```python
class RateLimitedAssistantAgent(AssistantAgent):
    """レートリミット付きのAssistantAgent"""
    def __init__(self, rate_limiter: RateLimiter, **kwargs):
        super().__init__(**kwargs)
        self._rate_limiter = rate_limiter

    async def on_messages(self, messages, cancellation_token):
        await self._rate_limiter.wait()
        return await super().on_messages(messages, cancellation_token)

    async def on_messages_stream(self, messages, cancellation_token):
        await self._rate_limiter.wait()
        async for item in super().on_messages_stream(messages, cancellation_token):
            yield item
```

議論の開始メッセージ（`task`パラメータ）はスレ立て風の文面とする:

```
スレタイ: {title}

{theme_description}

議論よろしく
```

返り値: `TaskResult` の `messages` リスト（各発言の `source` と `content` を含む）。

**ステップ 3-4: まとめエージェントの構築と実行**

- 対象: `src/agents/summarizer.py`
- 内容: 議論ログを受け取り、まとめサイト風の構造化データを出力するまとめAIエージェントを実装する。
- 理由: 議論に参加したエージェントとは独立した別のAIが、客観的にまとめを作成する。
- 依存関係: ステップ 2-2, ステップ 3-3
- リスク: 中（まとめ出力のフォーマット安定性）

まとめエージェントのシステムプロンプト:

```
あなたは2ちゃんねる（5ちゃんねる）のまとめサイト管理人です。
以下の掲示板の議論ログを「まとめサイト記事」として編集してください。

【出力フォーマット（JSON）】
{
  "title": "スレッドタイトル（キャッチーに編集して良い）",
  "category": "カテゴリー名",
  "thread_comments": [
    {
      "number": レス番号(int),
      "name": "投稿者名",
      "id": "投稿者ID",
      "content": "レス本文",
      "is_highlighted": 注目レスかどうか(bool),
      "highlight_color": ハイライト色 "red" or "blue" or null
    }
  ],
  "editor_comment": "管理人のコメント（記事冒頭に表示される短いリード文）",
  "reactions_summary": "読者の反応まとめ（記事末尾用テキスト）"
}

【編集ルール】
- 全レスを含める必要はない。面白いレス、重要なレスを選んでピックアップする
- 特に面白いレスや核心をつくレスには is_highlighted: true を付ける
- レスの順序は元の議論順を基本とするが、読みやすいように並べ替えても良い
- スレッドタイトルは元のテーマを活かしつつ、まとめサイト風にキャッチーに編集する
- 管理人コメントは短くまとめる
```

まとめエージェントは `AssistantAgent` を1体作成し、`team.run()` ではなく単体で `agent.run()` を呼ぶ。`task` にはフォーマット指示と議論ログ全文を渡す。

JSON出力の安定化: モデル側で `json_output=True` が使える場合はそれを利用する。使えない場合は、出力テキストから `{...}` 部分を正規表現で抽出するフォールバック処理を入れる。

スレタイ自動生成: ユーザーがスレタイ自動生成をONにしている場合、まとめエージェントが議論開始前にテーマからスレタイを生成する単独タスクを実行する。このスレタイを議論の `task` に渡す。

## リスクと対策

**リスク1: 2ch風の口調が安定しない**
対策: システムプロンプトに具体的なレス例（few-shot）を複数含める。改善されなければ、プロンプトの末尾に「以下は過去のレスの例です：」として3〜5件のサンプルレスを追加する。

**リスク2: まとめエージェントのJSON出力が壊れる**
対策: JSON抽出に正規表現フォールバックを実装する。`json.loads` が失敗した場合、`re.search(r'\{.*\}', text, re.DOTALL)` で JSON 部分を抽出する。それでも失敗したら、生テキストをそのまま表示する。

**リスク3: Gemini / Ollama / LM Studio のOpenAI互換APIが完全ではない**
対策: `model_info` パラメータで各モデルの機能（`function_calling`, `json_output`, `vision`）を明示的に指定する。問題が出た場合、OpenAI APIにフォールバックするオプションを用意する。

**リスク4: 会話数が多い場合のAPIコスト・レイテンシー**
対策: UIの会話数スライダーに注意書き（「会話数が多いほどAPIコストと生成時間が増加します」）を表示する。ステータスバーで進捗表示（「10/50レス完了」等）を行い、ユーザーの不安を軽減する。

**リスク5: AutoGen AgentChat のバージョン変更で API が変わる**
対策: `pyproject.toml` でバージョンを `>=0.4.0,<1.0` に固定する。`uv.lock` ファイルをGit管理に含め、再現可能なビルドを保証する。

**リスク6: Gradio `gr.HTML` でのCSSスコープ衝突**
対策: まとめ用CSSの全セレクタに `.matome-container` をプレフィクスとして付与し、Gradio本体のスタイルと衝突しないようにする。
